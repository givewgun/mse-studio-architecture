Rags to [Ad]Riches : ADR#3 Implement message queue and Gateway service in front of existing backend

Created by Gun Kaewngarm, last modified on Jun 28, 2024

Context: 

Adrich plans to expand its market presence by introducing approximately 2 million new smart labels, transitioning from sending minimal essential data to streaming raw sensor data directly to its backend system. This shift will significantly increase data volume and involve around 10,000 concurrent users, potentially overloading the current processing capacity of our backend infrastructure. Addressing this, we face the challenge of managing the increased data throughput efficiently to prevent the Backend processing service in the backend from overloading and maintain overall system stability without impacting other backend operations.

Quality Attributes related:

  • QA_2 - Performance 

Decision:

To address the potential overload issue, we will implement a Limit Event Response tactic using a message queue. This decision involves queuing incoming sensor data streams to regulate the flow into the processing system, ensuring that the Backend processing service in the backend can predictably process the data and will not overload when there is a spike.

Status:

Proposed

Consequences:

  • Predictable Backend processing: helps manage data flow effectively, predictably and constantly when there is a spike in data coming in.

  • Improved System Stability: With controlled data input rates, the overall backend system stability is enhanced, preventing crashes due to sudden spikes in data.

  • Improve Scalability: By queuing raw data stream, we can also scale out the Backend processing service where the data in the queue will be consumed by multiple Backend processing consumers which can reduce the overall time to process all data in the queue.

  • More effort  in Setup and Management: Implementing and maintaining a message queue introduces more effort to the engineer for setting up and managing the message queue.

  • Cost: Potential costs are associated with adopting a message queue and the operational overhead.
