Rags to [Ad]Riches : ADR #2 Implement Simulator in Middleware

Created by Gun Kaewngarm, last modified on Jun 28, 2024

Context:Â 

Adrich currently separates the data processing into another service called middleware, which will handle all of the necessary data transformations by processing it or delegating the task and calling an external api service to process the data, which then will be sent back to middleware to be transformed and send it back to backend API. We would like to implement a feature extractor here as it will reuse the existing architecture which offers modularity and high modifiability as engineers can configure which middleware to use for a specific product.

However, this method involves one service calling another and sometimes engaging in a chain of three or more services to compute and return the final result to the initiating service. This process leads to considerable communication overhead, significantly slowing down the system. In the new system, we plan to use the middleware without calling external API to reduce communication overhead while preserving the maintainability and Modifiability.

Quality Attributes related:

  • QA - Maintainability

  • QA - Modifiability

Decision:

We will implement a feature extractor in the middleware without calling an external service to run it. The new middleware will handle fetching a simulator executable database and running it, as there is little overhead when fetching the executable and running it, as it is very lightweight.

Status:

Agreed

Consequences:

  • Simplified Architecture and Easy for maintainance: Reusing existing architecture will make the system easier to maintain.

  • Improve Modifiability: Consolidation may compromise the modifiability of the system by combining multiple responsibilities into a single service, potentially making future changes more complex.

  • Easy transition: By reusing the existing architecture and the same operation flow, it will be easier for moving the data pipeline to a new one for new label

  • Increased Latency: By calling middleware, there will be some overhead instead of directly running the simulator in the backend API
