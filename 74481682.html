Rags to [Ad]Riches : ADR #1: Integrate Changes within Existing System

Created by Daniel Hales, last modified by Gun Kaewngarm on Jun 28, 2024

Context: 

The original construction was naive due to a lack of context from code discovery. Our analysis, which considered the positives of building a separate service and the potential risks, was thorough. We found that while the new approach offers benefits, it could also incur technical debt for future changes.

 Cons of the old approach:

  • To create a separate service, old functionalities from the previous architecture/design would need to be pulled into the new service

  • Implementing a new and separate service would require significant changes upstream and downstream to existing production systems

  • If Adrich were to deprecate the new service in the future (for any reason) this would incur substantial costs/resources to modify

  • • If we were to make our service extremely efficient, serping our services would not improve performance, as the bottleneck may then be elsewhere in the system.

Quality Attributes related:

  • QA - Maintainability / Operability

  • QA - Modifiability

  • QA - Performance

 

Decision:

The new data pipeline will be built on top of the existing architecture without significant changes. The performance improvement is to move into phase 2 of implementation, where the whole system needs to be refactored to achieve performance quality attributes instead of the new data pipeline. The proposed architecture and changes can be found in the revise architecture document.

Status:

Agreed

 

Consequences:

  • Ease of modification/Modularity: The current Adrich architecture is designed with modification in mind, using an intermediary service called Middleware. With this architecture, adding new functionality doesn’t require significant changes.

  • Easy maintenance/operation: The new data pipeline and calibration dashboard will be implemented on top of the existing architecture, so maintenance doesn’t require extra time to manage a separate service.

  • Reduce complexity: Since the system is built on top of the existing architecture, and we add a new component (Lego block) to the filter, the system's complexity remains the same without new additional services.

  • Ease of migration: Adrich has a plan to migrate the service to a new refactored architecture to improve performance in the future, so with the new changes behaving like the existing component, it will be easy for them to migrate this alongside the existing implementation later.

  • Decrease in performance: Adding more middleware (intermediary process) might decrease performance. However, as stated in the quality attributes, we anticipated that the data would still be processed within 30 seconds.

  • Require more time to understand the existing code base: We need more time and budget to understand the existing codebase as we have to integrate our changes on top of it.
